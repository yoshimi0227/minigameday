## 仮説と観測のズレ / 実装で踏んだ罠 (本体)

1. **[新機能] scale-to-zero は SSM automation で 2 回踏んだ** — いずれもユニットテスト・cdk-nag では捕まらない実機限定の不具合:
   - `documentArn` に `automation-definition/` 形式を渡して `The 'documentArn' parameter value is not valid`。正しくは **`document/` 形式**の ARN (CLI 例で確認)
   - 実験ロールに inline で ssm 権限を絞ったが `AuthorizationFailure`。ECS/RDS ロールと同じく**マネージドポリシー `AWSFaultInjectionSimulatorSSMAccess` + `iam:PassRole`** で解決
2. **想定どおり**: `maxDuration: PT1M` で実験は数秒で completed になり、fault の 5xx が停止条件 (5xx>10) に達する前に終わる (停止条件で誤って止まらない)。desiredCount=0 は残り、自己回復しない
3. **想定どおり**: 2 ラウンド表示 — スコア表がラウンド見出し + 小計でグループ化され、ラウンドフィルタで絞れる。round は採点ロジックに影響しない (表示のみ)
4. **R1/R2 の非対称性が意図どおり**: R1 は復旧 40 点が待てば入る (自己回復) ので判断の質は伝達点で見る。R2 は復旧点が人の対応速度に連動する

## 学び

- 「自己回復しない障害」は FIS ネイティブアクションだけでは作れない。SSM Automation を `aws:ssm:start-automation-execution` で起動し、**復元ステップを入れない**のが肝。これで MTTR が初めて「人が動いた時間」を測る
- 本番の GameDay と同じ「観察 → 対応」の段階設計が、既存の scenarioId フィルタ機構の再利用だけで実現できた
- クロススタックのイベント集約は固定名テーブルの名前インポートで cross-stack Export を作らずに成立する
